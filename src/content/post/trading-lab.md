## はじめに

普段、自分の自動売買やバックテストまわりはターミナル中心で扱っています。その場で見て、その場で動かして、そのまま直せるので、自分にはいちばん自然なやり方です。

今回作った Trading Lab は、そのバックテスト側をあえて Web UI に置き直してみたものです。

バックテスト自体は以前から普通に動いていました。ただ、ライブの運用側とは違って毎日見るものではありません。少し間が空くと、どこから見て、どの順で走らせて、何を確認するのかを思い出す必要があり、それが再開の負荷になっていました。

そこで今回は、普段ターミナルで扱っている流れを、そのまま画面として前に出したらどうなるのかを試してみました。派手なものを作りたいわけでも、公開向けに整えたいわけでもありません。しばらく触っていなかったバックテストを、もう一度触りやすい形につなぎ直すことが目的でした。

実際の Demo ページは <a href="https://trading-lab.pages.dev/" target="_blank" rel="noopener noreferrer">こちら</a> です。別ウィンドウで開きます。

## なぜ Web UI にしたのか

TUI は軽く、処理の実行と確認が一直線につながるので、開発や運用にはかなり強い道具です。普段の作業がターミナル中心になるのも自然だと思っています。

ただ、その強さがそのまま研究用の道具としての使いやすさにつながるとは限りません。バックテストは、ライブ運用のように毎日同じ熱量で触るものではないからです。久しぶりに開くと、どこまで進んでいて、次に何を見ればいいのかを頭の中で補いながら追う必要がありました。

そこで今回は、コマンドを覚えている前提ではなく、画面を開いた時点で流れが見える形に寄せてみることにしました。どこを先に見て、どこに注意して、次にどこを触るのかを最初から置いておけば、研究に入るまでの負荷を下げられると考えたからです。

## 構成

今回意識したのは、バックテストの本体を別物にしないことでした。

裏側ではもともと Python のスクリプト群とローカルファイルを中心に処理が回っています。そこに FastAPI を挟み、必要な状態や実行結果を取れるようにし、フロント側を React + Vite で組みました。

大事だったのは、GUI の都合で計算やデータの持ち方まで作り直さないことです。バックエンド側に残すべきものはそのまま残し、前に出したい部分だけを UI として切り出す。その形なら、もともとターミナルで扱っていたものの性格を崩さずに、触り方だけを変えられます。

本番の自動売買と違って、こちらは研究用の道具です。即時性を最優先する必要はありません。その代わり、途中で止まっても状態を追いやすいことと、また開いてみようと思えることを優先しました。

## 作って見えたこと

実際に作ってみていちばん大きかったのは、手順の見え方が変わったことでした。

ターミナルのときは、処理は動いていても流れそのものは自分の頭の中に置いていました。どの段階まで進んだか、次に何をするか、出力を見るならどこかを、その都度思い出しながらたどっていたわけです。

Web UI にすると、それがコマンドの列ではなく、状態の流れとして見えるようになります。どこまで進んでいるのか、今どこを見ているのか、次に触るのはどこかが、最初から配置された形で見える。ターミナルを置き換えたというより、その前にあった思い出す作業を減らせた感覚に近いです。

もうひとつ大きかったのは、エラーや停止箇所の追いやすさでした。細かい調査にはターミナルが必要でも、少なくともどの段階で止まったのか、次にどの処理を見るべきかはかなり掴みやすくなりました。入口が整理されるだけで、バックテストとの距離はかなり縮まります。

今回あらためて感じたのは、ファイルベースの処理でも十分に Web UI の対象になるということです。裏側が DB かどうかは本質ではありません。ローカルファイル中心でも、状態や進行や失敗箇所をまとめて見せることはできますし、どこを先に見て、次に何を触るかという流れも作れます。

以前は、フロントの価値を「見せること」や「整えること」に寄せて考えがちでしたが、今回はそこが違いました。見た目を整えることより、判断しやすい入口を作ることのほうが重要だったからです。バックエンドで動いている処理を、人が触って考えやすい形に置き直す。その役割のほうが、自分にはずっと大きく感じられました。

## Trading Lab という名前にした理由

名前を Trading Lab にしたのは、完成した運用画面ではなく、試したり見直したりする場として扱いたかったからです。

バックテストには、変えたくない部分と、試しながら見ていきたい部分があります。本番とつながる土台はそのまま残しつつ、試すための入口だけを扱いやすくする。その形が自分にはちょうどよさそうでした。

自分にとって必要だったのは、毎日張り付くための画面ではなく、また開いて考えられる研究用の場でした。Trading Lab は、そのための道具として作っています。

## おわりに

Trading Lab は、ターミナルで扱っていたバックテストを、そのまま Web UI に置き直してみた試みです。

今回確かめたかったのは、TUI で自然に回していたものを、重くしすぎずに研究しやすい形へ移せるかどうかでした。結果として、裏側の処理をそのまま活かしながら、再開しやすい入口を作る形は十分に成立しました。

普段は TUI で回しつつ、必要な場面だけ軽い GUI を持つ。その組み合わせは、自分のバックテストにはかなり相性がよさそうです。ターミナルで動かしていたものを、もう少し立体的に触れるようにする。その違いによって、処理を回すための道具が、研究に入りやすい道具へ少し変わりました。

技術的な詳細やコードは [GitHub](https://github.com/yktsnet/trading-lab) にまとめています。
