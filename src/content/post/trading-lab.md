## はじめに

普段、自分の自動売買やバックテストまわりはターミナル中心で扱っています。開発の流れとしても自然ですし、その場で見て、その場で動かして、そのまま直せるので、自分にとってはかなり馴染んだやり方です。

今回作った Trading Lab は、そのバックテスト側をあえて Web UI に寄せてみたものです。

もともとバックテスト自体は普通に動いていました。ただ、ライブの運用側とは違って毎日見るものではないので、少し間が空くと手順を忘れやすくなります。どこから見て、どの順に走らせて、どこを確認するのかを思い出す必要があり、そのうち少しずつ敬遠するようになっていました。

それなら、普段ターミナルで使っているものを、そのまま Web UI に寄せたらどうなるのだろうと思いました。派手にしたいとか、公開向けに整えたいとか、そういう話ではありません。しばらく触っていなかったものを、再実装しながらもう一度つなぎ直す機会にもなるはずでした。

今回は、Trading Lab をどう作ったかというより、ターミナルで扱っていたバックテストを Web UI に寄せてみて、何が見えてきたのかを書きます。

## TUIで使っていたものを、なぜGUIにしてみたのか

TUI の良さはかなりはっきりしています。軽いですし、開発や運用の流れとも噛み合います。ログもそのまま見られますし、処理の実行と確認が一直線につながっています。開発者が普段使う道具として、自然に感じる理由はよくわかります。

ただ、その自然さがそのまま、研究用の道具としての使いやすさにつながるとは限りません。バックテストはライブ運用とは違って、いつも同じテンションで触るものではないからです。久しぶりに開くと、どこを先に見ればよいのか、どこまで進んでいて、次に何をすればよいのかを、自分の頭の中で補いながら追う必要がありました。

ここで気になっていたのは、TUI が悪いということではありません。むしろ、TUI は処理を扱うにはとても強い道具です。ただ、情報の置かれ方はどうしても平面的になりやすく、流れや関係を直感的に掴むには少し工夫が要ります。毎日触っているものなら問題になりにくくても、間が空くものではその差が出やすいように感じていました。

だから今回は、見たい流れや気にしたい状態を、最初からこちらで配置できる GUI に寄せてみることにしました。コマンドを覚えている前提ではなく、画面を開いた時点で流れが見えるようにする。何を先に見てほしいのか、どこに注意を向けてほしいのかを、こちらで意図して置いてみる。その方が、研究に入るまでの負荷を下げられるのではないかと思ったのです。

## 構成をどう考えたか

今回の実装で意識したのは、バックテストの本体を別物にしないことでした。

もともと裏側では、Python のスクリプト群とローカルファイルを中心に処理が回っています。そこに FastAPI を挟み、必要な状態や実行結果を取れるようにして、フロント側は React + Vite で組みました。構成としてはそれだけです。

大事だったのは、GUI の都合で計算やデータの持ち方まで作り直さないことでした。バックエンド側に寄せるべきものはそのまま残し、前に出したい部分だけを UI として切り出す。その形なら、本来ターミナルで扱っていたものの性格を崩さずに、触り方だけを変えられます。

本番の自動売買と違って、こちらは研究用の道具です。だから、即時性を最優先する必要はありません。その代わり、途中で止まっても状態を追いやすいこと、また開いてみようと思えることの方が重要でした。

## Web UI にして見えたこと

実際に作ってみていちばん大きかったのは、手順の見え方が変わったことでした。

ターミナルのときは、処理は動いていても、流れそのものは自分の頭の中に置いていました。どの段階まで進んだか、次に何をするか、出力を見るならどこか、といったことを、その都度思い出しながらたどっていました。

Web UI にすると、それがコマンドの列ではなく、状態の流れとして見えるようになります。どこまで進んでいるのか、今どこを見ているのか、次に触るのはどこかが、最初から配置された形で見えるようになりました。ターミナルを置き換えたというより、ターミナルの前にあった思い出す作業を減らせた感覚に近いです。

もうひとつよかったのは、エラーや停止箇所の追い方でした。もちろんターミナルのすべてをそのまま置き換えられるわけではありませんし、画面から見えない部分もあります。それでも、少なくともどの段階で止まったのか、どの処理の様子を見るべきかはかなり辿りやすくなりました。ターミナルに近い感覚で異常に気づける入口を、画面側に作れたのは大きかったです。

このあたりは、実際に手を動かしてみて、思っていた以上に効いていました。しばらく触っていなかったバックテストをもう一度前に出し、構造を思い出しながらつなぎ直すには、かなり相性がよかったように思います。

## バックエンド寄りのものを前に出してみて見えたこと

今回の学びとして大きかったのは、ファイルベースの処理でも十分に Web UI の対象になると実感できたことでした。

最初は、こういうものはターミナルで扱うのが普通だと思っていましたし、実際その通りの場面も多いです。特に開発者にとっては、軽くて速くて、直接触れる TUI はかなり強い道具です。ただ、その強さがそのまま常に最適ということにはならないのだと、今回はあらためて感じました。

重要だったのは、裏側が DB かどうかではありませんでした。ローカルファイルを扱っていても、状態や進行や失敗箇所をまとめて見せることはできますし、どこを先に見て、どこに注意して、次に何を触るかという流れも作れます。つまり、バックエンド寄りの処理を、そのまま人が触りやすい形に前へ出すことはできる、ということです。

ここで効いていたのが、GUI の立体性でした。TUI は情報を連続的に追うには向いていますが、GUI は、見てほしい場所や触る順番を意図して置きやすい。今回のように、研究用の流れをもう少し入りやすくしたい場面では、この違いがかなり大きかったです。

以前は、フロントの価値をどうしても「見せること」や「整えること」に寄せて考えがちでしたが、今回はむしろ違いました。見た目そのものより、判断しやすい入口を作ることに価値がある。バックエンドで動いている処理を、人が触って考えやすい形に置き直す。その役割のほうが、自分にとってはずっと大きかったです。

## Labとしてどういう道具にしたかったか

名前を Trading Lab にしたのは、完成した運用画面というより、試したり見直したりする場として扱いたかったからです。

バックテストには、変えたくない部分と、試しながら見ていきたい部分があります。本番とつながる土台はそのまま残しつつ、試すための入口だけを扱いやすくする。その形が、自分にはちょうどよさそうでした。

自分にとって必要だったのは、毎日張り付くための画面ではなく、また開いて考えられる研究用の場でした。Trading Lab は、そこを狙って作っています。

## おわりに

Trading Lab は、普段ターミナルで扱っていたバックテストを、そのまま Web UI に寄せてみた試みです。

今回おもしろかったのは、バックテストを GUI にしたこと自体ではありませんでした。TUI では自然に回していたものを、GUI でも重くしすぎず、研究しやすい形に置き直せたことのほうが大きかったです。

TUI は軽く、開発や運用にはかなり強い道具です。ただ、それだけが常に最適というわけでもありません。特に、毎日見続けるものではなく、考えたり振り返ったりするための道具であれば、GUI にして流れや状態を意図して配置する意味はかなりあります。

しかも、それは必ずしも重いアプリケーションである必要はありません。裏側の処理をそのまま活かしながら、必要なところだけを前に出す形でも十分に成立します。普段は TUI で回しつつ、必要な場面だけ軽い GUI を持つ。今回の Trading Lab は、そのやり方がちゃんと成り立つことを確かめる試みでもありました。

ターミナルで動かしていたものを、もう少し立体的に触れるようにする。その違いによって、処理を回すための道具が、研究に入りやすい道具へ少し変わったように思っています。
