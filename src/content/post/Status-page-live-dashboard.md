## はじめに

ykts.net に Status ページを追加しました。自身で運用しているシステムのメトリクスを、5分ごとに更新して表示する小さなダッシュボードです。

この記事では、実装手順の話はしません。書きたいのは「なぜこれを作ったのか」「なぜポートフォリオサイトの中にわざわざ動くページを入れたのか」という背景のほうです。

私の自動売買システムそのものの設計やロジックについては、また別の記事で書くつもりです。今回はあくまで Status ページが完成したことと、そのページを作るまでの考え方についてのまとめになります。

技術的な仕様が気になる方は [portfolio-astro (GitHub)](https://github.com/yktsnet/portfolio-astro) から。

## 背景 ─ 半年かけて作ってきたものの延長として

去年の2025年7月、コードを1行も書いたことがない状態から、GPTとの壁打ちだけで自動売買のシステムを作り始めました。

最初は本当に何もわかっていませんでした。バックテストとは何か、どこまでを自動化すべきか、エントリーまでつなぐには何が必要か。その都度 GPT に聞いて、返ってきたものを読んで、動かして、壊して、また聞く。その繰り返しでした。

半年かけて、バックテストからエントリーまで一貫して回る仕組みを少しずつ形にしていき、1月頃にひとまず完成と言えるところまで持っていきました。いま動いているメインのシステムも、その延長線上にあります。

その頃には、システムの派生や新しい技術の導入も考え始めていましたが、同時に個人で閉じて作るだけではなく、チーム開発にも興味が出てきました。

そこで GitHub のリポジトリを公開し、名刺を作り、ポートフォリオサイトも立ち上げました。

## ポートフォリオサイトの中に、なぜ動くページを入れたのか

サイトを作る過程で Astro の SSG や SSR を知りました。記事のような静的なページだけでなく、必要なところだけ動的にできる。その仕組みというか組み合わせ自体がまず面白かったです。

せっかくなら、ただ作品一覧を並べるだけではなく、今動いているものをそのまま見せたいと思いました。

私のメインの開発はバック側です。日々見ているのはサーバーの中のログや状態で、表からは見えません。でも、裏側で動いているものをフロントにつなげると、一気に伝わり方が変わります。テキストで説明するより、実際に更新される画面を見せたほうが早い。

だから Status ページは、単なる飾りではなく Demo ページです。ライブで動いているものを、どう視覚的に見やすく、どう面白く見せるか。そのために作りました。

## ダッシュボードは、まず見てもらえないと意味がない

正直に言うと、今の Status ページそのものを、見ていてすごく面白い完成形だとは思っていません。自分のデータをかなりマスクしたうえで出しているので、見せられるものにも限界があります。

ただ、ダッシュボードそのものは、人が直感的に感じる見た目がまず大事だと思っています。意味がきれいに整理されていることも大切ですが、それ以前に「見たい」「また訪れたい」と思わせる入口がないと、そもそもページとして成立しません。

ライブで動いているものを見せるなら、数字の意味を丁寧に説明する前に、まず動いている気配が伝わることのほうが重要です。色や配置、更新される感じ、ぱっと見の印象。そういうものが先にあって、そのあとで「これは何のデータか」に入っていく。その順番はかなり意識しています。

今回の Status ページは、その考え方を自分のシステムで試してみた最初の実例です。

## 難しいものを作るだけでなく、つなげて見せるところまでやりたかった

自動売買をシステムとして作る難しさは、構築するロジックだけではありません。最初の頃は小型シングルボードコンピュータ Raspberry Pi を複数台使って役割を分けながら回していました。実際にやってみると、戦略のロジック以上に、環境構築や運用の再現性を保つことのほうがずっと難しいと感じる場面が多くありました。

その延長で、再現性を重視して NixOS に寄っていきました。単にプログラムを書くというより、どういう環境で、どういう依存関係で、どう安定して動かし続けるかをちゃんと作るようになっていった感覚です。

だから私が伝えたいのは、単に自動売買をやっているということではありません。難易度の高いバックエンドを作ることに留まらず、環境構築や運用、エッジ側の発想、さらにフロントに見せるところまで含めて、全体をつなげて考えているということです。

Status ページは、その発想を外から見える形にしたかったから作りました。

## 構成はかなりシンプルにした

Status ページの構成自体はシンプルです。サーバー側で定期的にデータをまとめて、Cloudflare 側に置き、SSR で読み出して表示する。それだけです。

```text
het (Hetzner VPS · NixOS · systemd timer)
  └─ Python → Cloudflare KV → Astro SSR → /Status
```

普段動いている側のデータを、そのまま公開用に整形して持ってくる。派手なことをしているわけではありません。むしろ重要なのは、構成そのものよりも「ライブで動いているものがそのまま見える」状態を作れたことでした。

## AI と一緒に作るという、自分の開発スタイル

じつはこのページも、ひとりで黙々とコードを書いて作ったわけではありません。作りたいものを言葉にして AI に伝え、返ってきたものを見て方向を直していく。それが私の開発スタイルです。

それが私の開発スタイルです。

もともと、考えていることを言葉にして人に伝えることには慣れていました。AI に対しても、やりたいことを構造的に説明して、返ってきたものを判断して方向を修正する、という進め方が自然にできました。

ちょうど去年の7月は、GPT を使って実装するというスタイルが一気に現実味を帯びてきた時期でもありました。自分にとっても、作りたいものを言葉で詰めていけば、実際にここまで形にできるのだと実感し始めたタイミングでした。

私はゼロから全部を自力で書ける人間ではありませんでした。でも、作りたいもののイメージはいつもありました。これとこれを繋いで、こうしたら上手くできるな、と構造をひとつひとつ言葉にし、ディレクションし、完成まで持っていくことはできるようになりました。

Status ページもそうでした。実装そのものは AI と一緒に進めていますが、このページを Demo にすること、ライブで動いているものを見せること、そしてそれをポートフォリオの中でどう位置づけるかを決めていたのは私です。

## 誠実さのためのページでもある

このページには、私の中でひとつはっきりした感覚があります。それは、説明だけするより、データを見せるほうが誠実だということです。

「こういうことができます」と口で説明することはできます。でも、実際に動いているものをそのまま見せられるなら、そのほうがずっと早いし、直感的だし何より刺さる。

だから私は、自動売買を単なる題材として終わらせず、そこからフロントにつなげるところまでやりました。

## まだ完成形ではなく、今は Demo である

とはいえ、今の Status ページが完成形だとは思っていません。

自分のデータをマスクして出しているので、見た目としても内容としても、まだ Demo の段階です。本当に面白くなるのは、これから実際にクライアントから仕事を受けて、扱うデータそのものを整理し、ライブでわかりやすく見せるところまで行ったときだと思っています。

今はその前段階です。まずは私自身のシステムで、「こういうことは実際にできる」と示している段階にあります。

## おわりに

Status ページは、完成のお知らせであると同時に、私の開発の方向を外に出したページでもあります。

去年7月から作ってきた仕組みを、今度はウェブサイトを通して、表に見える形までつなげてみました。

Status ページはその最初のデモです。まだ途中ではありますが、ライブで動いているものを、見える形にして外に出した。そのこと自体にこのページを作った意味があると思っています。
