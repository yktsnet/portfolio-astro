## はじめに

ykts.net に [Status ページ](https://ykts.net/Status/) を追加しました。自身で運用しているシステムのメトリクスを、5分ごとに更新して表示する小さなダッシュボードです。

この記事で書きたいのは実装手順ではありません。書きたいのは、なぜこれを作ったのか、そしてなぜポートフォリオサイトの中に、わざわざ動くページを置いたのかという背景です。

私の自動売買システムそのものの設計やロジックについては、また別の記事で書くつもりです。今回はあくまで Status ページを作った理由と、それを自分のサイトの中にどう位置づけているかをまとめます。

技術的な仕様が気になる方は [portfolio-astro (GitHub)](https://github.com/yktsnet/portfolio-astro) から確認できます。

## 背景 ─ 半年かけて作ってきたものの延長として

2025年7月、私はコードを1行も書いたことがない状態から、GPTとの壁打ちだけで自動売買のシステムを作り始めました。

最初は本当に何もわかっていませんでした。バックテストとは何か、どこまでを自動化すべきか、エントリーまでつなぐには何が必要か。その都度 GPT に聞いて、返ってきたものを読み、動かし、壊し、また聞く。その繰り返しでした。

半年ほどかけて、バックテストからエントリーまで一貫して回る仕組みを少しずつ形にし、2026年1月頃にひとまず完成と言えるところまで持っていきました。いま動いているメインのシステムも、その延長線上にあります。

作っていく中で見えてきたのは、難しいのは売買ロジックだけではないということでした。どういう環境で動かすのか、どう依存関係を揃えるのか、どう再現性を保つのか、どう安定して運用し続けるのか。そうした部分の比重が、想像していた以上に大きかったです。

その流れで、システムの派生や新しい技術の導入を考えるようになり、同時に、個人で閉じて作るだけでなくチーム開発にも関心を持つようになりました。そこで GitHub のリポジトリを公開し、名刺を作り、ポートフォリオサイトも立ち上げました。

Status ページは、その流れの中で生まれたものです。バック側で積み上げてきたものを、今度は外から見える形に変えてみたかった。その延長として作りました。

## ポートフォリオサイトの中に、なぜ動くページを入れたのか

サイトを作る過程で Astro の SSG や SSR を知りました。記事のような静的なページだけでなく、必要なところだけ動的にできる。その組み合わせ自体がまず面白かったです。

せっかくなら、作品一覧や記事を並べるだけで終わらせたくありませんでした。今この瞬間も動いているものがあるなら、それをそのまま見せられる形にしたいと思いました。

私の主な開発領域はバック側です。普段見ているのは、サーバーの中にあるログや状態、定期処理の結果やメトリクスです。そこには確かに面白さがありますが、そのままでは外から見えません。

だからこそ、裏側で動いているものをフロントにつなげる意味があると感じました。言葉で説明するだけでは伝わりにくいものでも、実際に更新されている画面として見えると、一気に伝わり方が変わります。

Status ページは、そのためのページです。単なる飾りではなく、私が普段バック側で扱っているものを、ウェブ上で見える形に変換した最初の実例です。

## ダッシュボードは、意味の前に入口が必要だと思っている

ダッシュボードというものは、意味が整理されていることだけでは成立しないと思っています。もちろん、数字の意味が明確であることは大事です。ただ、それ以前に、まず見てもらえること、ぱっと見たときに何かが動いていると感じられることが必要です。

公開されたページは、内部ツールとは違います。最初から文脈を共有している人だけが見るわけではありません。だからこそ、最初の数秒で「なんとなく見ていたくなる」「更新されている」「何か生きている」と感じてもらえる入口が必要だと考えています。

ライブで動いているものを見せるなら、数字の意味を丁寧に説明する前に、まず動いている気配が伝わることのほうが重要です。色、配置、更新される感じ、ぱっと見の印象。そうしたものが先にあり、そのあとで「これは何のデータか」に入っていく。その順番はかなり意識しています。

正直に言えば、今の Status ページそのものを、見ていてすごく面白い完成形だとは思っていません。公開できるようにかなりマスクしたデータを使っているので、見せられるものには限界があります。それでもなお、まずはこの考え方を自分のシステムで試してみることに意味があると思いました。

## 難しいものを作るだけでなく、つなげて見せるところまでやりたかった

自動売買をシステムとして作る難しさは、ロジックそのものだけではありません。最初の頃は Raspberry Pi を複数台使って役割を分けながら回していましたが、実際にやってみると、戦略の中身以上に、環境構築や運用の再現性を保つことのほうが難しいと感じる場面が多くありました。

その延長で、再現性を重視して NixOS に寄っていきました。単にプログラムを書くというより、どういう環境で、どういう依存関係で、どう安定して動かし続けるかまで含めて作るようになっていった感覚です。

だから私が伝えたいのは、単に自動売買をやっているという一点ではありません。難易度の高いバックエンドを作ること、環境を揃えること、運用を続けられる形にすること、そして最後に、それを外から見える形にすること。そうした全体をつなげて考えているということを示したかったのです。

Status ページは、その発想を表に出したかったから作りました。裏で完結している仕組みを、裏のまま終わらせず、見えるところまで持ってくる。私にとって重要だったのはその部分でした。

## このページは、小さくても「動いて見える」ことを優先している

Status ページの中でやっていること自体は、発想としてはそこまで複雑ではありません。裏側で定期的に更新される状態を、公開用に整えたうえでウェブ上に反映させているだけです。

```
het (Hetzner VPS · NixOS · systemd timer)
  └─ Python → Cloudflare KV → Astro SSR → /Status
```

ただ、この形にしたのには理由があります。欲しかったのは、常時接続の重い仕組みや大げさなリアルタイム基盤ではなく、数分単位で更新される状態を、安定して外に見せられることでした。今回の用途では、そのくらいの粒度で十分でした。

もうひとつ大きかったのは、公開のためにデータを一度整える前提を入れられることです。普段動いている側の情報をそのまま外へ出すのではなく、見せてよい形に変えてから扱う。そうすることで、実際に動いている気配を残しつつ、公開ページとしての整理もしやすくなります。

このページで重要だったのは、構成を必要以上に複雑にすることではなく、裏側で更新されているものが、きちんと表でも更新されて見える状態を作ることでした。小さくても、その往復が成立していることに意味があると考えています。

## これは完成形ではなく、第一段階としての Demo である

とはいえ、今の Status ページを最終形だとは考えていません。

今見せているのは、自分のデータをかなりマスクしたうえで構成した、最初のデモです。見た目としても内容としても、まだ発展の余地は大きいです。ただ、それは弱みというより、このページの位置づけそのものです。

今回やりたかったのは、完成したプロダクトを見せることではありません。バック側で動いている仕組みを、ウェブ上で見える形に変換し、そこまでを一続きで扱えることを示すことでした。つまり、このページはゴールではなく、そこへ向かうための最初の段階です。

本当に面白くなるのは、今後、クライアントワークの中で扱うデータや文脈に合わせて、何を見せるべきかを整理し、それをライブでわかりやすく見せるところまで進んだときだと思っています。今回はその手前として、まず自分自身のシステムで、その入口を形にしました。

## おわりに

[Status ページ](https://ykts.net/Status/) は、単なる完成のお知らせではありません。私がこれまでバック側で積み上げてきたものを、今度は外から見える形につなげてみた、その最初のページです。

2025年7月から作ってきた仕組みは、最初はサーバーの中だけで完結していました。そこから少しずつ、環境や運用も含めて形を整え、最後にウェブサイトを通して表に見えるところまで持ってきました。

まだ途中ではありますが、裏で動いているものを、外から見える形にした。そのこと自体に、今回このページを作った意味があると思っています。
