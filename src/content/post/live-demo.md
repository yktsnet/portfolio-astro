## はじめに

Hetzner VPS 上で動いている自動売買パイプラインの状態を、Cloudflare KV 経由で集約し、Astro SSR で表示する公開ページを作りました。[Live Demo](https://ykts.net/live-demo/) として公開しています。

自分がこれまで主に作ってきたのは、サーバーの中で動くバックエンドや運用の仕組みでした。ただ、それらはそのままだと外から見えにくく、ポートフォリオとしても伝わりにくいところがあります。

そこで、実際に動いている自動売買パイプラインを題材に、機密を保ったまま、運用の気配や設計の考え方が伝わる公開ページを作りました。この記事では、Live Demo の実装手順そのものよりも、なぜこれを作ったのか、何をどう見せることにしたのか、その判断の背景を書きます。

技術的な実装が気になる方は [portfolio-astro (GitHub)](https://github.com/yktsnet/portfolio-astro) から参照できます。

## バックエンドを外に出す必要があった

私は長く経営に関わってきましたが、その後、個人開発者として自分で実装するようになりました。2025年7月には、コードを1行も書いたことがない状態から、GPT と対話しながら自動売買システムを作り始めました。

半年ほど集中して取り組む中で、バックテストからエントリーまで一貫して回る仕組みを少しずつ形にしていき、今年の初め頃にひとまず完成と言えるところまで持っていきました。その過程で強く感じたのは、難しいのは売買ロジックだけではなく、環境構築や再現性、状態管理、安定運用のほうでもあるということでした。

そのあと、個人で完結する開発だけではなく、チーム開発の場にも進みたいと考えるようになりました。そこで必要になったのが、自分が何を作ってきたのか、何ができるのかを外に伝えることでした。

ただ、バック側で動いているものは、そのままだと伝わりにくく、自動売買のような題材は機密性も高いため、中身をそのまま出すことはできません。そこで、何をどの粒度で見せれば、自分のスキルが伝わるのかを考えて作ったのが Live Demo です。

## 自動売買という特殊性

自動売買は、ポートフォリオの題材としては少し特殊です。実際に動いている仕組みである一方で、戦略そのものや損益、銘柄、判断ロジックの詳細は、そのまま外に出したいものではありません。

ただ、その周辺には、バック側で積み上げてきた仕事があります。定期実行、状態の集約、遅延の把握、データの整形、外に出せる形への変換。そうした、サーバーの中では確かに重要な部分を、ポートフォリオとして意味が通る形にしたいと考えました。

内部では、実際の実行や制御を担う側があり、その周辺でデータ整備、監視、状態集約、可視化用の加工を担う側があります。今回の Live Demo で前面に出しているのは、そのうち後者に近い情報です。内部そのものを見せるのではなく、内部で動いている仕組みの一部を、外から見ても意味が通る形に翻訳することを意識しました。

## なぜ公開ページか

ポートフォリオとして何かを見せる方法は、文章や図解だけではありません。実際に動いているものを見せるほうが早い場面があります。特に、自分が扱ってきたものがバックエンドや運用のような見えにくい領域であれば、その傾向はなおさらです。

一方で、内部情報をそのまま出してしまうと、本来守るべきものまで外に出てしまいます。そこで必要だったのは、内部のデータをそのまま公開することではなく、公開可能な形に整えた別の面を作ることでした。

Live Demo は、そうした意味での公開ページです。内部状態の鏡ではなく、公開用に再構成したサマリーであり、何が動いているかを全部説明するためのページではありません。外から見ても、どういう種類の仕組みを作ってきたのかが伝わることを重視しました。

## 公開する指標をどう選んだか

外から見たときにまず「何かが動いている」と感じられることを重視して、Live Demo では SESSION、HEATMAP、LATENCY、FRESHNESS という4つの指標を置いています。

ここで出しているのは、損益や売買判断そのものではなく、システムが動いていること、どの時間帯に活動があるか、どのくらいの遅延で反応しているか、元データの鮮度がどうか、といった運用の輪郭です。

この構成にしたのは、公開ページとして意味が通る情報だけを残したかったからです。内部ではもっと多くの情報を扱っていますが、それらを全部見せる必要はありませんでした。チャートに円が並んでいれば今日も動いていることがわかる。遅延が大きく跳ねていれば、どこかで状態が変わっていると感じられる。そのくらいの粒度で十分に伝わるものを選んでいます。

## 構成は薄く

構成自体はかなりシンプルです。

```text
het (Hetzner VPS · NixOS · systemd timer)
└─ Python → Cloudflare KV → Astro SSR → /live-demo
```

サーバー側で定期的に状態を集約し、公開用に整えたうえで、Astro SSR で読み出して表示しています。

今回ほしかったのは、秒単位で張り付くようなリアルタイム性ではありませんでした。数分単位で更新される状態を、安定して外に見せられることのほうが大切でした。そのため、公開ページでは5分の遅延を挟み、機密性のある情報は落としたうえで、必要な指標だけを KV に送っています。

このページは内部状態のミラーではありません。内部で扱うデータをそのまま出すのではなく、公開面として成立するように整えた別レイヤの表示です。見せるために単純化しているのではなく、見せるために作り直している、という感覚に近いです。

## 技術より先にあったこと

技術的には、Cloudflare KV、Astro SSR、Hono、SVG と比較的軽い構成です。ライブラリを重ねて派手な画面を作るより、依存を増やさずに必要な情報だけを安定して返すことを優先しました。

たとえば SVG チャートは `document.createElementNS` で描画しています。ここでやりたかったのは、可視化ライブラリを使いこなすことではなく、公開ページとして必要な表現だけを、自分で制御しやすい形で持つことでした。

ただ、今回の制作で本当に重要だったのは、技術選定そのものより、どこまでを公開用の面として切り出すかの判断です。内部運用の情報を、意味が通る最小単位まで削り、見せられる形に整える。その設計のほうが、このページでは本質でした。

## おわりに

[Live Demo](https://ykts.net/live-demo/) は、内部で動いている自動売買パイプラインの一部を、公開可能な粒度へ整えて、ポートフォリオとして伝わる形にしたページです。

私がやりたかったのは、自動売買そのものを公開することではなく、見えにくいバックエンドの実装や運用を、機密を守りながら、外から見ても意味が通る面に変換することでした。

その意味で今回の制作は、バックエンド、運用、可視化を別々に考えるのではなく、何をどう見せれば自分のスキルが伝わるかまで含めて設計した最初のデモになりました。
